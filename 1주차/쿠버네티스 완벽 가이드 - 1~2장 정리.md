## 1장 도커 복습과 Hello, Kubernetes

### 1.1 도커 복습

쿠버네티스는 도커 외의 런타임도 지원하므로 도커를 깊게 알 필요는 없지만, **도커 이미지 지식**(설계 → Dockerfile → 빌드 → 레지스트리 푸시)은 필수다.

#### 1.1.1 도커 컨테이너란?

- 도커 컨테이너 = **도커 이미지를 기반으로 실행되는 프로세스**
- 이미지만 있으면 어떤 환경에서든 동일하게 동작 → **BORA**(Build Once, Run Anywhere) — 자바의 WORA에 빗댄 표현
- VM 대비 **가볍고 시작/중지가 빠름**: VM은 하이퍼바이저 위에 게스트 OS를 통째로 띄우지만, 컨테이너는 호스트 커널을 공유하며 네임스페이스 분리 + cgroups 제어로 독립 환경을 구성 → 게스트 OS 부팅 대기가 없음

#### 1.1.2 도커 컨테이너 설계 4원칙

1. **1 컨테이너 = 1 프로세스** — 한 이미지에 여러 프로세스를 올리면 에코시스템과 맞지 않고 관리가 어려움
2. **변경 불가능한 인프라(Immutable Infrastructure)** — "환경은 새로 만들고, 만든 환경은 절대 수정하지 않는다". 기동 후 외부에서 바이너리/패키지를 받아오면 외부 요인에 따라 결과가 달라지므로, 필요한 리소스는 이미지 안에 포함
3. **경량 이미지** — 노드에 이미지가 없으면 풀(다운로드)해야 하므로 가볍게. 방법: 패키지 캐시 삭제, 멀티 스테이지 빌드, 경량 배포판(Alpine/Distroless), 레이어 축소·스쿼시
4. **root 이외 실행 계정** — root 실행은 보안 사고 위험. 최소 권한 계정 사용(실무에선 전용 계정 권장)

#### 1.1.3 도커 파일(Dockerfile) 작성법

기본 흐름: `FROM` 기반 이미지 → `RUN`/`COPY` 처리 → 기동 명령 정의.

```dockerfile
FROM golang:1.14.1-alpine3.11        # 기반 이미지
COPY ./main.go ./                    # 파일 복사
RUN go build -o ./go-app ./main.go   # 빌드 시 실행
USER nobody                          # 실행 계정
ENTRYPOINT ["./go-app"]              # 기동 시 실행 명령
```

**기반 이미지 비교 (표 1-1)**

| 이미지 | 크기 | 특징 |
|---|---|---|
| scratch | 최소 | 아무것도 없음, 셸도 없어 디버그 어려움 |
| alpine | 작다 | musl libc + busybox. glibc와 동작 차이 주의 |
| distroless | 작다 | 특정 런타임(Python/Node 등)만 포함 |
| ubuntu / centos | 크다 | 범용 배포판 |
| UBI | 크다 | 레드햇 지원, 기술 지원 필요 시 |

**ENTRYPOINT와 CMD**: 기동 시 `$ENTRYPOINT $CMD`가 실행된다고 이해하면 됨. 둘 다 기동 옵션으로 덮어쓰기 가능. 관례: ENTRYPOINT엔 바꿀 일 없는 부분, CMD엔 기본 인수(예: ENTRYPOINT=`/bin/sleep`, CMD=`3600`).

| ENTRYPOINT | CMD | 실행 결과 |
|---|---|---|
| "/bin/ls" | "-a" | `/bin/ls -a` |
| "/bin/ls" | 없음 | `/bin/ls` |
| 없음 | "/bin/ls" | `/bin/ls` |
| "/bin/sh", "-c" | "ls -a" | `/bin/sh -c "ls -a"` |

**주요 명령 (표 1-3)**: FROM(기반 이미지) / LABEL(메타데이터, MAINTAINER는 비추천) / USER(실행 계정) / WORKDIR(작업 디렉터리) / EXPOSE(Listen 포트) / COPY(파일 복사) / ADD(tar.gz 해제 후 복사) / RUN(빌드 시 명령 실행) / ENTRYPOINT·CMD(기동 시 명령/인수)

#### 1.1.4 도커 이미지 빌드

```bash
$ docker image build -t sample-image:0.1 .            # -t 이름:태그(보통 버전)
$ docker image build -t sample-image:0.2 -f Dockerfile-MultiStage .  # 파일 지정
$ docker image ls                                     # 확인
```

**멀티 스테이지 빌드** — golang 이미지엔 컴파일 도구(~364MB)가 포함돼 결과 이미지가 커짐(377MB). 컴파일용/실행용 컨테이너를 나누고 **결과물만 복사**해 해결:

```dockerfile
# Stage 1: 컴파일
FROM golang:1.14.1-alpine3.11 as builder
COPY ./main.go ./
RUN go build -o /go-app ./main.go

# Stage 2: 실행용 (결과물만 복사)
FROM alpine:3.11
COPY --from=builder /go-app .
ENTRYPOINT ["./go-app"]
```

- 효과: 377MB → **13.1MB**(alpine), scratch 기반이면 **7.4MB**
- 실행 이미지에 불필요한 소프트웨어가 없어 **보안에도 유리**. 3단계 이상 구성 가능, BuildKit으로 병렬 빌드도 가능

#### 1.1.5 이미지 레이어 통합·축소

- **Dive**: 레이어별 파일 변경/용량을 조사하는 도구
- `docker image build --squash`: 레이어를 하나로 합쳐 축소(집필 시점 실험 기능)

#### 1.1.6 도커 레지스트리로 이미지 업로드

- 레지스트리 = 이미지 보관 서버 (Docker Hub, GCR, Amazon ECR)
- **'Docker Official Images' 공식 이미지 사용 권장**. 공식도 취약할 수 있어 실서비스는 **Trivy, Clair** 등 보안 스캔 검토
- 이름 형식: `레지스트리호스트명/네임스페이스/저장소:태그` (Docker Hub는 호스트명 생략, 네임스페이스=사용자명)

```bash
$ docker login
$ docker image tag sample-image:0.1 DOCKERHUB_USER/sample-image:0.1
$ docker image push DOCKERHUB_USER/sample-image:0.1
$ docker logout
# 처음부터 푸시할 이름으로 빌드해도 됨
$ docker image build -t DOCKERHUB_USER/sample-image:0.1 .
```

#### 1.1.7 컨테이너 기동

쿠버네티스 사용 시에는 쿠버네티스가 기동하므로 직접 쓸 일은 적다.

```bash
$ docker container run -d -p 12345:8080 sample-image:0.1
$ curl http://localhost:12345
Hello, Kubernetes
```

## 2장 왜 쿠버네티스가 필요할까?

### 2.1 쿠버네티스란?

- **컨테이너화된 애플리케이션의 배포·확장 관리를 자동화하는 플랫폼**(컨테이너 오케스트레이션 엔진)
- 도커 단독으로는 여러 호스트 통합 관리가 불가 → 일정 규모 이상에선 오케스트레이션 엔진 필요
- 경쟁(도커 스웜, 아파치 메소스)이 있으나 쿠버네티스가 **사실상 표준**(점유율 약 77%)
- 런타임은 도커 외 containerd, cri-o 등 사용 가능
- 용어: 컨테이너가 실제 기동하는 머신 = **노드**, 노드를 관리하는 노드 = **마스터**

### 2.2 쿠버네티스의 역사

- 구글 내부 도구 **Borg**에서 아이디어를 얻은 오픈 소스. 2014년 6월 공개 → 2015년 7월 v1.0과 함께 **CNCF**로 이관
- CNCF 성숙도: **Graduated > Incubating > Sandbox** — 쿠버네티스는 Graduated
- 표준이 된 배경 = 대형 클라우드의 관리형 서비스:
  - **GKE** (Google) — 2014.11 최초 출시, 2015.8 GA
  - **AKS** (Azure) — 2017.2 출시, 2018.6 GA
  - **EKS** (AWS) — 2017.11 출시, 2018.6 GA

### 2.3 쿠버네티스를 사용하면 무엇을 할 수 있을까?

컨테이너 실서비스 운영 과제(노드 관리, 스케줄링, 롤링 업데이트, 스케일링, 모니터링, 복구, 서비스 디스커버리, 로드 밸런싱, 데이터/로그 관리 등)를 자동화하거나 간단한 오퍼레이션으로 해결.

| 기능 | 핵심 내용 |
|---|---|
| **선언적 코드 관리(IaC)** (2.3.1) | YAML/JSON **매니페스트**로 컨테이너와 주변 리소스 관리. VM 시대의 CloudFormation/Heat에 대응 |
| **스케일링/오토 스케일링** (2.3.2) | 같은 이미지의 레플리카를 여러 개 배포해 부하 분산·다중화. 부하에 따라 레플리카 수 자동 증감 |
| **스케줄링** (2.3.3) | 컨테이너를 어느 노드에 배치할지 결정. 어피니티/안티 어피니티로 제어(예: 디스크 I/O 많은 컨테이너 → SSD 노드), 멀티존 분산 배치 |
| **리소스 관리** (2.3.4) | 노드의 CPU/메모리 여유에 따라 자동 스케줄링. 클러스터 오토 스케일링으로 노드 자동 추가/삭제 |
| **자동화된 복구(self-healing)** (2.3.5) | 프로세스 정지 감지 시 자동 재스케줄링·재배포. 노드 장애에도 서비스 영향 없이 복구. HTTP/TCP·셸 스크립트 헬스 체크 설정 가능 |
| **로드 밸런싱·서비스 디스커버리** (2.3.6) | **서비스/인그레스**가 로드 밸런서 기능 제공. 확장 시 자동 등록/삭제, 장애 시 분리, 롤링 업데이트 시 사전 분리 자동. 마이크로서비스 간 참조에 서비스 디스커버리가 유용 |
| **데이터 관리** (2.3.7) | 백엔드 스토어로 **etcd** 채용(클러스터 구성으로 이중화). 설정·인증 정보를 안전하게 중앙 관리 |

**매니페스트 예제 (코드 2-1)**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: sample-app
  template:
    metadata:
      labels:
        app: sample-app
    spec:
      containers:
        - name: nginx-container
          image: nginx:1.16
```

**에코시스템 연계 예**: Ansible·Spinnaker(배포), Fluentd(로그), GitLab(CI/CD), Jenkins(잡), Prometheus(모니터링), Spark(잡 네이티브 실행), Kubeflow(ML), Rook(분산 스토리지), Vitess(MySQL 클러스터) 등. 기능 확장 구조가 있어 쿠버네티스를 프레임워크 삼아 자체 플랫폼 구현도 가능.

### 2.4 정리

VM 시절엔 자동 배포를 위해 CloudFormation 작성, 로드 밸런서 구조 구축, 업데이트 구조 수정을 반복해야 했지만, 쿠버네티스는 **사전 개발된 자동화 기능**을 그대로 쓸 수 있다. 컨테이너의 이동성·경량성을 활용한 빠른 개발 + 배포 자동화 + 배포 스크립트 등을 에코시스템과 공용으로 쓸 수 있는 것이 큰 장점.
