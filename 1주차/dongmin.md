## 1. 한눈에 보는 결론

Docker는 컨테이너 이미지를 만들고 실행하는 데 강하다. 하지만 컨테이너 수가 많아지고 서버가 여러 대가 되면 배치, 복구, 확장, 로드 밸런싱, 설정 관리, 데이터 관리가 어려워진다. Kubernetes는 이 문제를 선언적 방식으로 관리하는 컨테이너 오케스트레이션 플랫폼이다.

이번 범위에서 가장 중요한 문장은 다음과 같다.

> 이미지는 버전으로 고정하고, 컨테이너와 Pod는 언제든 다시 만들어질 수 있다고 생각해야 한다.

## 2. 먼저 던질 질문

아래 질문에 답할 수 있으면 이번 범위의 핵심은 잡힌다.

- 컨테이너는 왜 VM과 다른가?
- 이미지를 새로 빌드했는데 기존 컨테이너가 그대로인 이유는 무엇인가?
- Dockerfile은 왜 단순 설치 스크립트가 아닌가?
- 컨테이너 내부에 저장한 파일은 왜 위험한가?
- Docker만으로 여러 서버의 컨테이너를 운영하면 무엇이 어려운가?
- Kubernetes는 Docker를 대체하는가, 아니면 다른 역할을 하는가?
- Pod IP에 직접 접근하면 왜 안 되는가?
- 자동 복구가 “장애 원인 해결”과 다른 이유는 무엇인가?

## 3. 개념 지도

```text
Docker
  -> Image: 실행 가능한 패키지
  -> Container: 이미지를 실행한 인스턴스
  -> Dockerfile: 이미지를 재현 가능하게 만드는 빌드 명세
  -> Registry: 이미지를 올리고 내려받는 저장소

Kubernetes
  -> Pod: 컨테이너 실행 최소 단위
  -> Deployment: Pod 복제 수 유지와 롤링 업데이트
  -> Service: 변하는 Pod 앞에 고정 접근 지점 제공
  -> Scheduler: Pod를 어느 Node에 둘지 결정
  -> Controller: 현재 상태를 원하는 상태로 맞춤
```

## 4. Docker 복습: 헷갈리기 쉬운 부분

### 4.1 컨테이너는 작은 VM이 아니다

컨테이너는 독립 OS를 부팅하는 VM이 아니라 호스트 커널을 공유하면서 프로세스, 파일시스템, 네트워크, 자원 사용량을 격리해 실행하는 방식이다. 그래서 컨테이너 안에서 Ubuntu를 쓰더라도 실제 커널은 호스트 커널이다.

이 차이를 모르면 “Docker 안에서 모든 OS를 다 실행할 수 있다”는 오개념이 생긴다. pfSense처럼 FreeBSD 커널 자체가 필요한 시스템은 Docker 컨테이너가 아니라 VM이나 물리 서버로 다뤄야 한다.

### 4.2 이미지는 설계도, 컨테이너는 실행 인스턴스다

이미지는 실행 가능한 패키지이고, 컨테이너는 그 이미지를 실제로 실행한 인스턴스다. 이미지를 새로 빌드했다고 이미 떠 있는 컨테이너가 자동으로 바뀌지는 않는다. 변경 사항을 반영하려면 새 이미지로 컨테이너를 다시 만들어야 한다.

운영에서는 다음 감각이 중요하다.

- 이미지는 버전으로 고정한다.
- 컨테이너는 언제든 삭제되고 다시 만들어질 수 있게 둔다.
- 중요한 데이터는 컨테이너 내부에 두지 않는다.

### 4.3 Dockerfile은 설치 스크립트가 아니라 이미지 빌드 명세다

Dockerfile의 각 명령은 이미지 레이어를 만들고 빌드 캐시와 이미지 크기에 영향을 준다. `RUN apt update`와 `RUN apt install`을 무작정 나누면 불필요한 레이어가 늘 수 있다.

PDF의 멀티 스테이지 빌드 예시처럼 빌드 도구가 필요한 단계와 실제 실행 이미지를 분리하면 이미지 크기와 공격 표면을 줄일 수 있다. 실행 컨테이너에는 “운영에 필요한 것만 남긴다”는 기준이 필요하다.

### 4.4 ENTRYPOINT는 편하지만 복잡해지면 장애 포인트가 된다

`ENTRYPOINT`는 컨테이너 시작 시 실행할 주 프로그램을 고정하는 성격이 강하고, `CMD`는 기본 명령이나 인자를 제공하는 성격이 강하다.

로컬 노트의 GPU 서버 컨테이너처럼 entrypoint에서 계정 생성, SSH 설정, Jupyter 실행, 권한 설정을 처리할 수 있다. 다만 시작 스크립트가 복잡해질수록 장애 원인을 찾기 어려워진다. entrypoint는 여러 번 실행되어도 망가지지 않게 만들고, 로그를 반드시 남겨야 한다.

### 4.5 컨테이너 내부 저장소는 영구 저장소가 아니다

컨테이너 삭제와 함께 내부 데이터가 사라질 수 있다. 사용자 홈, DB 데이터, 학습 결과, 로그는 볼륨이나 바인드 마운트로 분리해야 한다.

특히 연구실 서버처럼 사용자별 UID/GID와 홈 디렉터리 권한이 중요한 환경에서는 컨테이너 내부 사용자와 호스트 파일 권한이 맞지 않으면 접근 제한이 깨질 수 있다.

### 4.6 레지스트리와 포트 매핑

레지스트리는 이미지 저장소이지 실행 환경이 아니다. 일반 흐름은 다음과 같다.

```bash
docker image build -t sample-image:0.1 .
docker image tag sample-image:0.1 USER/sample-image:0.1
docker image push USER/sample-image:0.1
```

`latest`는 편하지만 같은 이름이 다른 이미지를 가리킬 수 있으므로 운영에서는 버전, 날짜, Git SHA처럼 재현 가능한 태그를 쓰는 편이 낫다.

포트 매핑은 `-p 2024:22`처럼 호스트 포트와 컨테이너 포트를 연결하는 것이다. 여기서 2024는 호스트 포트이고 22는 컨테이너 내부 포트다. Kubernetes로 넘어가면 이 개념이 Service, NodePort, LoadBalancer, Ingress로 확장된다.

## 5. Kubernetes가 필요한 이유

### 5.1 Docker만으로 운영할 때 생기는 문제

Docker만으로 컨테이너 하나를 실행하는 것은 쉽다. 문제는 컨테이너 수가 많아지고 서버가 여러 대가 되는 순간부터다.

- 어느 서버에 어떤 컨테이너를 띄울지 결정해야 한다.
- 컨테이너가 죽었을 때 다시 띄워야 한다.
- 트래픽을 여러 컨테이너에 나누어야 한다.
- 버전 업데이트 중 장애를 줄여야 한다.
- 설정, 비밀값, 데이터 저장 위치를 관리해야 한다.

Kubernetes는 이런 운영 문제를 자동화하기 위한 컨테이너 오케스트레이션 플랫폼이다.

### 5.2 Kubernetes는 Docker 대체품이 아니다

Docker는 이미지를 만들고 컨테이너를 실행하는 도구에 가깝다. Kubernetes는 여러 노드 위에서 컨테이너 기반 애플리케이션을 배치, 복구, 확장, 노출, 업데이트하는 운영 시스템이다.

실제 Kubernetes에서는 Docker 대신 containerd 같은 컨테이너 런타임을 많이 쓴다. 그래도 Kubernetes는 결국 컨테이너 이미지와 런타임 위에서 동작하므로 Docker 기본기는 필요하다.

### 5.3 핵심은 선언적 상태 관리다

Docker에서는 `docker run`으로 지금 컨테이너 하나를 띄우는 흐름이 익숙하다. Kubernetes에서는 YAML 매니페스트에 원하는 상태를 적고 `kubectl apply -f`로 적용한다.

예를 들면 다음처럼 생각한다.

- “nginx 컨테이너 하나 실행”이 아니라 “nginx Pod 3개를 유지하라”
- “이 포트 열어줘”가 아니라 “이 라벨의 Pod 묶음을 Service로 노출하라”
- “죽으면 내가 다시 실행”이 아니라 “컨트롤러가 원하는 상태로 되돌리게 하라”

## 6. Kubernetes 운영 개념

### 6.1 Pod

Pod는 Kubernetes의 최소 실행 단위지만 컨테이너 하나와 같은 말은 아니다. Pod 안에는 하나 이상의 컨테이너가 들어갈 수 있고, 같은 Pod의 컨테이너는 네트워크와 일부 스토리지를 공유한다.

중요한 점은 Pod가 영구적인 대상이 아니라는 것이다. Pod는 죽고 다시 만들어질 수 있으며 IP도 바뀔 수 있다. 그래서 사용자는 Pod IP에 직접 의존하지 말고 Service를 통해 접근해야 한다.

### 6.2 Deployment

Deployment는 Pod를 안정적으로 유지하고 업데이트하는 리소스다. Pod를 직접 만들면 장애 복구, 복제 수 유지, 롤링 업데이트를 직접 챙겨야 한다.

실무에서는 “Pod를 띄운다”보다 “Deployment로 원하는 Pod 집합을 선언한다”가 기본 흐름이다.

### 6.3 스케일링과 오토 스케일링

스케일링은 서버 한 대를 키우는 것만 뜻하지 않는다. 같은 이미지를 기반으로 여러 Pod 복제본을 늘려 부하를 나누는 수평 확장이 핵심이다.

다만 오토 스케일링이 애플리케이션 병목을 자동으로 해결해 주지는 않는다. 애플리케이션이 상태를 로컬 디스크나 메모리에만 들고 있으면 복제본을 늘려도 장애가 날 수 있다. 상태 저장 위치와 세션 처리 방식까지 함께 설계해야 한다.

### 6.4 스케줄링과 리소스 관리

Scheduler는 노드의 CPU, 메모리, 제약 조건, affinity/anti-affinity 등을 보고 Pod를 어디에 둘지 결정한다. PDF에서도 디스크 I/O가 많은 컨테이너를 SSD 노드에 배치하는 예처럼 워크로드 특성에 맞는 배치가 가능하다고 설명한다.

리소스 관리는 장애 예방의 핵심이다.

| 항목 | 의미 | 오개념 주의 |
| --- | --- | --- |
| `requests` | 스케줄링 기준이 되는 최소 보장량 | 실제 사용량 제한으로 착각하면 안 됨 |
| `limits` | 넘지 못하게 제한하는 상한 | 너무 작으면 OOMKilled나 성능 저하 발생 |

### 6.5 자동 복구

자동 복구는 Kubernetes가 깨진 Pod를 다시 만들거나 다른 노드로 옮기는 기능이다. 하지만 원인을 고치는 기능은 아니다. 애플리케이션 자체가 계속 죽으면 Kubernetes는 계속 재시작할 뿐이고 `CrashLoopBackOff`가 날 수 있다.

장애가 나면 다음을 같이 본다.

- `kubectl logs`
- `kubectl describe`
- readiness/liveness probe
- `requests`/`limits`
- 최근 이미지 변경 여부

### 6.6 로드 밸런싱과 서비스 디스커버리

Service는 라벨 셀렉터로 대상 Pod 묶음을 찾고 고정된 이름과 접근 지점을 제공한다. 같은 클러스터 안에서는 DNS 이름으로 서비스를 찾을 수 있고, 외부 공개는 NodePort, LoadBalancer, Ingress로 처리한다.

Service는 단순 포트 포워딩이 아니라 “변하는 Pod 집합 앞에 고정 주소를 세우는 리소스”로 이해해야 한다.

### 6.7 데이터 관리

Pod가 다시 만들어질 수 있다는 말은 Pod 내부 파일시스템에 중요한 데이터를 두면 사라질 수 있다는 뜻이다. 유지되어야 하는 데이터는 PersistentVolume, 외부 스토리지, 별도 DB 서비스로 분리해야 한다.

etcd는 Kubernetes 클러스터 상태 저장소이지 애플리케이션 데이터를 넣는 일반 DB가 아니다.

## 7. 명령어 루틴

처음에는 모든 리소스를 외우기보다 아래 흐름을 손에 익힌다.

```bash
kubectl get pods
kubectl get nodes -o wide
kubectl apply -f app.yaml
kubectl describe pod <pod-name>
kubectl logs <pod-name>
kubectl exec -it <pod-name> -- /bin/bash
kubectl delete -f app.yaml
```

Docker Desktop이나 Minikube는 운영용 클러스터가 아니라 실습용 로컬 클러스터에 가깝다. 운영 클러스터는 노드 구성, 네트워크, 스토리지, 인증, RBAC, 모니터링, 백업까지 별도로 설계해야 한다.

## 8. 오개념 방지표

| 오개념 | 바로잡기 |
| --- | --- |
| 컨테이너는 작은 VM이다 | 컨테이너는 호스트 커널을 공유하는 격리된 프로세스 실행 환경이다 |
| 이미지를 새로 빌드하면 컨테이너도 바뀐다 | 기존 컨테이너는 그대로다. 새 이미지로 다시 만들어야 한다 |
| 컨테이너 안에 저장하면 계속 남는다 | 컨테이너 내부 저장소는 사라질 수 있다. 볼륨/외부 스토리지가 필요하다 |
| `latest`는 최신이라 안전하다 | 어떤 이미지인지 재현하기 어렵다. 운영에서는 고정 태그가 낫다 |
| Kubernetes는 Docker 대체품이다 | Kubernetes는 컨테이너 운영/관리 시스템이다 |
| Pod IP에 접속하면 된다 | Pod IP는 바뀔 수 있다. Service로 접근해야 한다 |
| 오토 스케일링이 성능 문제를 해결한다 | 상태 관리와 애플리케이션 구조가 나쁘면 복제본을 늘려도 문제가 난다 |
| 자동 복구가 원인을 고친다 | Kubernetes는 재시작/재배치할 뿐 원인 분석은 별도로 해야 한다 |
| etcd에 앱 데이터를 넣으면 된다 | etcd는 클러스터 상태 저장소다 |

- [ ] 이미지와 컨테이너의 차이를 설명할 수 있다.
- [ ] Dockerfile의 레이어와 멀티 스테이지 빌드가 왜 중요한지 말할 수 있다.
- [ ] `ENTRYPOINT`가 복잡할 때 생기는 운영 리스크를 설명할 수 있다.
- [ ] Docker 포트 매핑 `-p 2024:22`의 의미를 설명할 수 있다.
- [ ] Docker만으로 여러 서버를 운영할 때의 한계를 말할 수 있다.
- [ ] Pod, Deployment, Service의 역할을 구분할 수 있다.
